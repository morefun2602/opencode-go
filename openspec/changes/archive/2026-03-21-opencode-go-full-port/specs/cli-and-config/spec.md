# 能力：cli-and-config

## ADDED Requirements

### Requirement: 与上游配置一致

对于两种实现中均存在的设置，系统应当使用与上游 OpenCode **相同的配置文件名**、**文件格式**（例如上游指定 JSON 或 YAML 时）以及**顶层与文档化键路径**。系统应当公布**所跟踪的上游兼容性引用**（版本或提交范围），说明上述一致性适用的范围。仅 Go 实现使用的设置必须使用上游未占用的键，或本规范参考附录中列出的已文档化命名空间前缀（实现可以增加链接自本文的附录文件）。

#### Scenario: 共享工作区配置可加载

- **当** 用户按兼容性引用，在预期搜索路径放置对上游 OpenCode 有效的配置文件
- **则** 本实现应当按下文相同优先级解析相同键，且对常用选项不要求单独文件

### Requirement: 配置优先级

系统应当按以下严格顺序应用配置（后者覆盖前者）：二进制内嵌默认值；已发现的配置文件中的值；环境变量；命令行 flag。具体环境变量名与 flag 名必须写在实现文档中，且在未标记 **BREAKING** 的同一主版本内必须保持稳定。

#### Scenario: 环境变量覆盖文件

- **当** 同一逻辑项同时在配置文件与本规范允许的环境变量中设置
- **则** 进程应当采用环境变量的值

### Requirement: CLI 帮助与用法错误

系统应当为根命令及每个会打印非成功用法的子命令提供 `-h` / `--help`。对用法或 flag 校验错误，系统应当以退出码 **2** 退出；成功打印帮助时应当以 **0** 退出。

#### Scenario: 非法 flag 退出码为 2

- **当** 用户传入未定义的 flag，或违反 flag 间互斥/依赖关系
- **则** 进程应当向标准错误输出诊断信息并以退出码 2 退出

### Requirement: 运行退出码

系统应当将分类错误映射到稳定退出码：**0** 成功；**2** 用法/校验；**1** 内部或未分类失败；前台命令被用户中断（SIGINT）时为 **130**（在平台可区分时）。额外码必须有文档说明。

#### Scenario: 运行中被中断

- **当** 用户发送中断信号以取消正在执行的前台命令
- **则** 进程应当以退出码 130 结束，除非平台无法区分中断与其他信号

### Requirement: 从 CLI 启动 HTTP 服务模式

系统应当提供已文档化的子命令或 flag 组合，用于启动 `http-api` 规范中定义的 HTTP API 服务，绑定地址与鉴权等参数来自同一套配置优先级规则。

#### Scenario: 启动服务与配置一致

- **当** 用户使用有效配置调用已文档化的 serve/start 类命令
- **则** HTTP 服务器应当按 `http-api` 要求监听，并在启动时记录生效的绑定地址
