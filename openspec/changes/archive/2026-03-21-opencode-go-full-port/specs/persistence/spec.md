# 能力：persistence

## ADDED Requirements

### Requirement: 会话状态持久存储

在启用持久化时，系统应当将会话元数据与消息历史写入持久本地存储，使得进程重启后按配置恢复范围内的会话。默认存储技术必须为 SQLite，除非经显式构建或配置覆盖且文档标注为 **BREAKING**。

#### Scenario: 重启后会话恢复

- **当** 某工作区启用持久化且进程正常关闭后再次启动
- **则** 系统应当重新加载该工作区下此前存储且未被显式删除的会话

### Requirement: Schema 迁移

系统应当为数据库 schema 维护版本，并在启动时自动应用向前迁移。若磁盘上 schema 版本高于二进制所支持的上限，系统必须拒绝启动并给出明确错误信息。

#### Scenario: 旧二进制拒绝新库

- **当** 数据库 schema 版本大于二进制支持的最大版本
- **则** 进程应当以非零状态退出并打印错误，提示升级或备份

### Requirement: 数据完整性

在存储支持事务时，更新关联行（例如消息与会话）的写操作必须在事务层面原子化。失败事务产生的部分写入不得使存储处于下一次启动迁移可检测的不一致状态。

#### Scenario: 失败 turn 不损坏历史

- **当** 写入用户与助手消息的事务在提交过程中失败
- **则** 两条消息均不得表现为已提交，或存储应当回滚到事务前状态

### Requirement: 工作区隔离

不同工作区根路径（或配置的工作区标识）的数据应当相互隔离：对某一工作区的查询不得返回属于另一工作区标识的行。

#### Scenario: 查询限定在工作区

- **当** 运行时为工作区 A 加载历史
- **则** 结果集应当排除存储在工作区 B 下的消息
