# 能力：llm-and-tools

## ADDED Requirements

### Requirement: 提供商抽象

系统应当将对语言模型提供商的出站调用隔离在内部接口之后，使得更换提供商实现时除配置外无需修改 CLI 或 HTTP handler。

#### Scenario: 由配置选择提供商

- **当** 配置指定某一受支持的提供商实现
- **则** 运行时应当将补全请求路由到该提供商，且除配置与提供商注册外无需改代码

### Requirement: 流式补全

当上游 API 支持流式时，系统应当消费流式分片并增量转发给会话消费者（CLI 或 HTTP），且不得在内存中缓冲完整补全，除非特定提供商限制已在发行说明中说明。

#### Scenario: 完成前可见部分输出

- **当** 某次补全启用流式
- **则** 消费者应当在流成功结束之前收到增量助手内容

### Requirement: 工具调用契约

系统应当以显式名称与经 schema 校验的参数调用已注册工具；校验失败时系统不得执行该工具，并应当向模型循环返回结构化错误，且符合 agent-runtime 错误处理路径。

#### Scenario: 非法工具参数被拒绝

- **当** 模型以未通过校验的参数请求工具
- **则** 系统不得执行该工具的副作用，并向编排层暴露校验错误

### Requirement: 工具与 MCP 失败语义

若工具或基于 MCP 的动作失败（网络错误、非零退出、超时），系统应当在日志中记录失败并带关联 ID，应当将结构化失败描述回传给智能体循环以便模型按策略继续或停止。系统不得静默忽略工具失败。

#### Scenario: 记录工具失败

- **当** 工具调用在已被接受后失败
- **则** 日志应当包含含工具名与会话 ID 的错误级别条目，且智能体循环应当收到与成功相区分的失败结果

### Requirement: 提供商出站 HTTP

对模型提供商的出站 HTTP 应当使用 Go 的 `net/http`（或薄封装），请求超时来自配置，并应当遵守上下文取消。

#### Scenario: 提供商请求超时

- **当** 提供商 HTTP 请求超过配置的超时时间
- **则** 客户端应当取消该请求并向调用方返回归类为超时的错误
